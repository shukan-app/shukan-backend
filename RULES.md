# Rules

## Implementation Rules

- 実装を追加・変更した場合は、必ずテストを実行し、すべてのテストケースがパスするかを確認する
- テストケースがパスしない場合、原因を調査し内容および修正方針を報告する (そのまま修正には進まない)
- 報告後、ユーザーの承認が得られた場合は修正を実施する
- 報告内容は `失敗テスト名`, `原因仮説`, `修正案` および `影響範囲` を含める
- エラー発生時には `Cloud Logging` を用いて特定可能な形でエラーを出力する (`Logback` + `SLF4J`)

## Coding Styles

- 暗黙的で短い記述よりも, 明示的で冗長な記述を優先する
- `{`, `}` は省略せずに必ず記述する (特に `switch` のブロックや1行の `if` 文に注意)
- `Nullable` なものは明示的に `org.jspecify.annotations.Nullable` アノテーションを付与する (基本は `NonNull`)

## Communications

- ユーザーの要求やドキュメントに曖昧な内容が含まれている場合は、放置せずに質問し内容を明確にしてから作業に進む
